# OI-0018 - Location-Scoped Production Printer Routing

## Status

Open

## Area

Printing / Devices / Menu Production Routing

## Summary

PLAN-0005 delivered the printing foundation: a generic outbox/work-item mechanism, a worker host,
ESC/POS receipt formatting, and one configured network printer endpoint for receipt printing. That
is valid for the foundation, but it is not enough for production venue operations.

Production use requires each `Location` to independently map order production components to its
own printers. Example routes include `drinks`, `kitchen`, `kitchen-deep-fried`, `dessert`, `coffee`,
and `bar`.

This issue tracks the backend/data-model and configuration work needed for production docket
routing. It is not implemented in PLAN-0005 and must remain open until a dedicated implementation
plan builds and verifies it.

## Human Decisions Recorded

- Production route codes are strings, not enums.
- Production route codes must be slug-like and contain no spaces.
- Allowed route-code pattern:

```text
^[a-z0-9][a-z0-9_-]{0,63}$
```

- Route examples:
  - `drinks`
  - `kitchen`
  - `kitchen-deep-fried`
  - `dessert`
  - `coffee`
  - `bar`
  - `cold-bar`
  - `hot-bar`
- For standard products, the production route should be stored on the product, for example
  `Product.ProductionRouteCode`.
- If a standard product has no production route, it does not produce a production docket and does
  not print to a kitchen/bar/dessert station.
- Back Office should eventually warn when a product has no production route and therefore will not
  print anywhere for production.
- Combo/grouped products require a separate production-component model. A combo may display as one
  sales line on a receipt but expand into multiple production components for dockets.

## Required Behaviour

- Printer routing must be scoped by `Location`.
- Each location configures its routing independently of every other location in the same tenant or
  organisation.
- Printer routing is data, not code.
- The system must not globally route `kitchen`, `drinks`, or any other route across all locations.
- A production docket must contain only the components relevant to that printer/station.
- Customer receipts remain whole-order documents.
- Production dockets are separate documents from customer receipts.
- Production dockets must not be generated by filtering a customer receipt client-side.

## Standard Product Example

| Product | Production route |
|---------|------------------|
| Coke | `drinks` |
| Beer | `drinks` |
| Burger | `kitchen` |
| Chips | `kitchen-deep-fried` |
| Cake | `dessert` |

For a normal product, the order line can snapshot the product route at sale time.

```csharp
public sealed class Product
{
    public Guid Id { get; set; }
    public string Name { get; set; } = "";
    public string? ProductionRouteCode { get; set; }
}
```

## Combo / Grouped Product Example

A combo may be sold and shown to the customer as:

```text
Fish & Chips Combo    $18.00
```

But production may need:

```text
KITCHEN
- Fish

FRYER
- Chips

BAR
- Drink
```

This means the domain must distinguish the customer-facing sales line from production components.

Recommended model concept for whichever plan implements this:

```csharp
public enum ProductCompositionType
{
    Standard = 0,
    Combo = 1
}

public sealed class ProductComponent
{
    public Guid Id { get; set; }

    // Parent combo/grouped product.
    public Guid ParentProductId { get; set; }

    // Actual product that gets produced.
    public Guid ComponentProductId { get; set; }

    public decimal Quantity { get; set; }

    // Optional override. If null, use ComponentProduct.ProductionRouteCode.
    public string? ProductionRouteCodeOverride { get; set; }
}
```

At order time, production output should be snapshotted separately from the receipt line:

```csharp
public sealed class OrderLineProductionComponent
{
    public Guid Id { get; set; }
    public Guid OrderLineId { get; set; }
    public string NameSnapshot { get; set; } = "";
    public decimal Quantity { get; set; }
    public string? ProductionRouteCodeSnapshot { get; set; }
}
```

## Missing Route Behaviour

Approved interim rule:

- If a product or production component has no production route, it does not print anywhere.
- The sale/order line is still valid.
- This is not a failed order.
- Back Office should later warn users that the product/component will not print to a production
  station.

Still undecided:

- What happens when a route exists but no printer is mapped for the location.
- What happens when a route points to a disabled printer.
- What happens when a route points to an unreachable printer.
- Whether missing route-printer mappings should block order send, warn only, or create failed
  print work items.

## Location Mapping Example

| Route | Location A printer | Location B printer |
|-------|--------------------|--------------------|
| `drinks` | Bar printer | Drinks printer |
| `kitchen` | Kitchen printer | Kitchen printer |
| `kitchen-deep-fried` | Fryer printer | Kitchen printer |
| `dessert` | Dessert bar printer | Kitchen printer |

## Scope Boundary

This OI is about production routing and docket generation.

In scope for the future implementation plan:

- Product production-route field.
- Combo/component production model.
- Location-scoped route-to-printer mapping.
- Production docket document model.
- Production docket outbox/work item creation.
- Worker processing of production print work.
- Back Office warnings/configuration once the model exists.

Out of scope for this OI:

- Printer discovery.
- USB transport.
- MAUI terminal implementation.
- Installer/daemon/update mechanics.
- Provider-specific hardware adapters.
- Stripe Terminal.
- Closing OI-0017.

## Recommended Implementation Plan

This should not be squeezed into PLAN-0006 Milestone A.

Recommended sequencing:

1. Keep PLAN-0006 PWA-first for shell, Back Office, terminal, payments, display, and minimal KDS.
2. Create a dedicated follow-up plan for production routing and docket printing before building a
   real KDS or real kitchen/bar printing workflow.
3. Let Back Office configure these routes only after the backend model exists.
