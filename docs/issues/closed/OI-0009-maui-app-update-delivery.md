# OI-0009 — MAUI App Update Delivery

## Status

Closed

## Area

Devices / Deployment

## Summary

How will Daxa Terminal (the .NET MAUI Windows POS app) receive and apply updates in production venues?

## Context

Daxa Terminal runs on Windows POS devices in venues. Unlike PWA (which auto-updates via the browser), a MAUI app is a native Windows application that requires explicit update delivery.

Venues may have limited IT support. POS terminal updates must be reliable, testable, and not interrupt trading hours. Some venues may want to control when updates are applied.

## Impact

- Determines the MAUI app distribution and update mechanism.
- Affects deployment documentation and venue IT guidance.
- Affects CI/CD pipeline (build artefacts).
- Affects testing workflow (signed builds required).

## Options

1. **MSIX package + Windows Package Manager (winget)** — Modern Windows distribution. Supports auto-update via MSIX. Requires code signing certificate.
2. **MSIX package + Daxa-hosted AppInstaller feed** — Daxa hosts an `.appinstaller` XML file. Windows can auto-check for updates. No store required.
3. **Windows Store (MSIX)** — Managed distribution. Subject to Store review process. May not be suitable for B2B POS.
4. **Custom updater (self-check on startup)** — App checks Daxa API for new version and downloads installer. Full control. More engineering effort.
5. **IT-managed deployment (Intune / SCCM / PDQ)** — Enterprise customers deploy via MDM. Requires MSIX or MSI packages. No auto-update for unmanaged venues.

## Recommendation

**MSIX + Daxa-hosted AppInstaller feed** for most venues. Enterprise customers can use MDM (Option 5). Requires a code signing certificate from the start.

## Decision Needed

- Update delivery mechanism for production MAUI app.
- Code signing certificate strategy.
- Whether updates are mandatory or deferred.

## Answer / Decision Input (2026-06-30)

The production update mechanism will be managed through the local Blazor SSR application running on the venue's Daxa Local server.

The local application will check a public Daxa update feed to determine whether a newer published version of Daxa Terminal is available. If an update is available, the operator can choose to start the update from the local administration/deployment UI.

The update is manually initiated. When the operator presses the update button, the local Blazor SSR application calls a daemon/tool running on the Linux Daxa Local machine. That daemon is responsible for connecting to each Windows POS device on the local network and performing the installation/update tasks.

This daemon-based deployment tooling is not required for the first MAUI application build. It will be built after-the-fact as part of production deployment tooling.

### Packaging

The likely packaging strategy is:

- MSIX package preferred where practical.
- Alternative installer packaging may be used if it is simpler for venue deployment.
- A Winamp-style installer approach may be considered because it is open source and easy to package.

The final packaging mechanism can remain implementation-dependent until the first production deployment path is tested.

### Code Signing

Daxa will use Microsoft's Azure code signing facility / Azure Trusted Signing where practical.

The signing strategy should cover:

- MAUI binary artefacts.
- MSIX package artefacts, if MSIX is used.
- Installer artefacts, if a separate installer is used.
- Release artefacts published to the public update feed.

Signed artefacts should be generated by CI/CD rather than signed manually on developer machines.

### Mandatory vs Deferred Updates

Updates are not mandatory by default.

Venue operators may choose whether to install updates. Daxa should not normally force updates because POS systems are operationally sensitive and venues may not want software changes during trading periods.

If a major security issue requires urgent action, Daxa will notify customers via email and the public website. Even then, the update remains operator-controlled unless a future managed-service contract or enterprise deployment model explicitly allows Daxa-managed mandatory updates.

### Implementation Notes

The update mechanism should support:

- Version discovery from a public feed.
- Displaying available update details in the local Blazor SSR UI.
- Manual approval before installation.
- Calling a privileged local daemon/tool on the Linux server.
- Updating one or more Windows POS devices on the local network.
- Recording update attempts, success, failure, and installed version.
- Avoiding automatic interruption during trading hours.

### Closure Criteria

This issue can be closed when:

- The chosen package format is confirmed.
- The public update feed format is documented.
- The local update UI flow is documented.
- The Linux daemon/tool responsibility is documented.
- The code signing approach is confirmed.
- The operator-controlled update policy is recorded in deployment documentation.

## Decision Addendum

OI-0009 is resolved.

Daxa Terminal updates will be operator-controlled and delivered through the Daxa Local / Daxa Hybrid management layer where local server infrastructure exists.

Cloud-only deployments may use the same update feed and package artefacts, but without relying on forced background updates.

## Decision

The production update mechanism for the Windows .NET MAUI Daxa Terminal app will use:

```text
Signed application package / installer
    ↓
Daxa public update feed
    ↓
Local Daxa server update UI
    ↓
Privileged local update daemon/tool
    ↓
Windows POS devices on the venue network
```

The local Blazor SSR / Back Office application will show available updates to authorised users.

An authorised operator can choose to start an update. The local server then calls a privileged daemon/tool on the Linux server. That daemon is responsible for connecting to Windows POS devices on the local network and applying the update.

## Packaging

The preferred packaging option is MSIX where practical.

Alternative installer packaging may be used if MSIX does not fit a specific deployment path.

The final package format may remain implementation-dependent until production deployment testing confirms the best option.

Acceptable package outputs may include:

- MSIX.
- MSI.
- EXE installer.
- Other signed Windows deployment package if justified later.

The application should not depend on Windows Store distribution for MVP.

## Code Signing

All production Windows app artefacts must be signed.

Code signing should cover:

- MAUI application binaries.
- MSIX packages, if used.
- Installer artefacts, if used.
- Update feed artefacts where applicable.

The intended signing approach is Azure Trusted Signing or an equivalent commercial code-signing process.

Signing should be performed by CI/CD and not manually on developer machines.

## Update Policy

Updates are deferred/operator-controlled by default.

Updates must not automatically interrupt trading.

The local administration UI should show:

- Current installed version.
- Available version.
- Release notes or change summary.
- Update size, where available.
- Target devices.
- Device online/offline status.
- Last update attempt.
- Update success/failure status.

A venue operator can choose when to apply updates.

Mandatory forced updates are not part of the default MVP model.

If a critical security issue requires urgent action, Daxa can notify customers through customer communications, website notices, and the admin UI. The update should still remain operator-controlled unless the customer's contract explicitly allows managed mandatory updates.

## Local Server Daemon Responsibility

The local daemon/tool should be responsible for:

- Discovering registered Windows POS devices.
- Checking device update eligibility.
- Copying or downloading the signed installer/package.
- Running the installer/update command.
- Reporting progress.
- Recording success/failure.
- Preserving logs for support.
- Avoiding updates during configured blackout/trading windows unless overridden.

This daemon/tool may be built after the first MAUI build and does not need to block early application development.

## Security Requirements

The update system must protect against unauthorised or tampered updates.

The production design should include:

- Signed artefacts.
- Trusted update feed.
- Versioned releases.
- Hash/checksum validation.
- Rollback path where practical.
- Audit logging.
- Role/permission checks before initiating updates.
- Environment/channel separation.

Release channels may include:

```text
Internal
Preview
Stable
Customer-specific, if required later
```

## Audit Requirements

Each update attempt should record:

- Who initiated the update.
- When it was initiated.
- Which location/server initiated it.
- Which device was targeted.
- Previous version.
- Target version.
- Result.
- Error details, if failed.
- Installer/package identifier.

## Consequences

This model keeps the update process practical for venues with limited IT support while avoiding unsafe automatic updates during trading.

It also keeps the architecture compatible with local, hybrid, and future managed deployments.

## Status Update

This open issue is resolved by accepting signed artefacts, a Daxa update feed, operator-controlled deployment, and a local server daemon/tool for applying updates to Windows POS devices.
