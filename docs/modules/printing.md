# Module: Printer Service

The printer service routes print work to receipt, production, and future label printers.

PLAN-0005 implemented the server-side receipt-printing foundation. Production routing and local
terminal-attached printing are separate concerns and remain deferred.

---

## Responsibilities

- Server-controlled network receipt printing.
- ESC/POS command generation.
- Cash drawer kick via printer port.
- Generic outbox/work-item processing for print work.
- Print retry handling.
- Reprint flow.
- Production docket routing for kitchen/bar/dessert stations (deferred, OI-0018).
- Label printer routing (later).

## Printer Types

| Type | Connection | Owner | MVP |
|------|------------|-------|-----|
| Receipt printer | Network (Ethernet/Wi-Fi) | Server / `DaxaPos.Workers` | Yes |
| Receipt printer | Local USB / locally attached Windows printer | Future MAUI terminal app | Deferred |
| Kitchen/bar printer | Network | Server / `DaxaPos.Workers` | Deferred - OI-0018 |
| Label printer | USB / network | Future hardware/device plan | Later |

Network receipt printing is handled by the server-side worker using the outbox mechanism. Local USB
or locally attached Windows printer support belongs to the future MAUI terminal app because the
printer is physically attached to that terminal, not the server. Production kitchen/bar routing is
tracked separately by OI-0018.

## Implementation Status

### Implemented In PLAN-0005

- Generic `OutboxWorkItem` mechanism.
- `DaxaPos.Workers` host.
- `OutboxProcessorWorker`.
- `PrintReceiptOutboxProcessor`.
- `OutboxRetryPolicy`.
- `EscPosReceiptFormatter`.
- `IPrinterTransport`.
- `NetworkPrinterTransport`.
- Network-only receipt printing to a single configured host/port.
- Cash drawer kick bytes when an order includes a cash payment.

Current receipt printing flow:

```text
Order completed
  -> domain event handler enqueues PrintReceipt outbox row
  -> DaxaPos.Workers polls pending work
  -> worker loads immutable order/payment/refund/tax snapshot data
  -> ReceiptRenderer creates ReceiptDocument
  -> EscPosReceiptFormatter creates bytes
  -> IPrinterTransport sends bytes to configured network printer
```

## Server vs MAUI Ownership

The server owns:

- Network printers reachable from the server/local network.
- Server-side worker/outbox printing.
- Receipt print work generated by completed orders.
- Future production docket work for kitchen/bar/dessert routing.

The future MAUI terminal owns:

- Local USB printers physically attached to the Windows terminal.
- Locally installed Windows printers available only to that terminal.
- Any terminal-local printing path that cannot be reached by the server worker.

This keeps server-side printing and terminal-local hardware access separate.

## Deferred: Location-Scoped Production Printer Routing

Tracked as [OI-0018](../issues/open/OI-0018-location-scoped-production-printer-routing.md).

Production venues need each `Location` to route order components to its own printers independently.
Examples:

- `drinks` -> bar printer
- `kitchen` -> kitchen printer
- `kitchen-deep-fried` -> fryer printer
- `dessert` -> dessert bar printer

Important distinction:

- Customer receipts are whole-order documents.
- Production dockets are station-specific documents.
- A production docket must include only the components relevant to that station.

For normal products, the production route should be stored on the product and snapshotted onto the
order. Combo/grouped products need separate production components so one receipt line can fan out to
multiple production stations.

Missing production route rule:

- If a product/component has no production route, it does not print anywhere.
- The sale remains valid.
- Back Office should eventually warn the user that the item will not print to production stations.

## Permissions

`printing.manage` is reserved but not implemented. It should be added only when a print-job
administration/retry endpoint exists.

Receipt reprint is handled by `receipts.reprint`, which was implemented in PLAN-0005.
