# Open Questions

## Product scope

- Is the first product aimed at hospitality only, or broader retail/services from day one?
- Which industry templates should exist at MVP launch?
- Should repair/service jobs be Phase 1 or Phase 3?
- Should bakery pre-orders be MVP or Phase 2?

## Payments

- Which AU payment provider should be integrated first: Tyro, Zeller, Square, Stripe Terminal, or Windcave?
- Should the first integrated payment be chosen for commercial market fit or developer speed?
- Should manual EFTPOS be the only MVP payment option initially?
- How will payment provider credentials be stored and rotated?
- What level of refund support is required in MVP?

## Tax

- Should AU/NZ be the only tax mode in MVP?
- Should US tax architecture exist in the model but not be exposed in UI?
- Should the app integrate with Avalara/TaxJar later for US taxes?
- Should tax settings be locked down by country templates?

## Offline

- How much offline selling should MVP support?
- Can card payment be disabled when offline unless the provider supports offline payments?
- Should local storage be SQLite?
- Should there be a local venue server later?

## Devices

- Which Windows POS hardware should be targeted first?
- Do we need USB printer support in MVP?
- Do we need cash drawer support in MVP?
- Do we support barcode scanners as keyboard-wedge input first?

## Reporting

- What are the mandatory reports for MVP?
- Should exports to CSV/Xero/MYOB be Phase 2 or Phase 3?

## Multi-location

- Should MVP support one venue only, with multi-location data model prepared?
- Should central catalogue and store overrides be Phase 3?
